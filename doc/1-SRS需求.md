# MyTodos 团队待办 App — 需求规格说明书（SRS）

## 1. 引言

### 1.1 文档目的

本文档用于描述“MyTodos 团队待办 App”的需求规格，包括功能需求、非功能需求、界面需求、数据库/数据需求与外部接口需求，为产品设计、开发实现、验收测试提供依据。

### 1.2 产品一句话需求

一个面向 2-5 人小团队的移动端待办事项协作应用。使用 Rust + Tauri + Vue 3 开发，通过 Gitee 代码片段（Gist）API 存储数据，无需自建后端数据存储。

### 1.3 范围与边界

- 目标用户：2-5 人的小团队（例如同学小组、家庭分工、创业小团队）。
- 目标端：移动端优先（Android/iOS），可选支持桌面端复用同一代码基座。
- 数据存储：远端使用 Gitee 代码片段 API；本地仅做缓存、离线队列、配置存储。
- 不在范围内：
  - 自建服务器、传统后端数据库、IM 即时通讯。
  - 复杂的组织/权限体系（例如多部门、多项目层级、细粒度 ACL）。
  - 企业级审计、SLA、可用性承诺。

### 1.4 术语与缩略语

- Workspace：工作区/团队空间，一组成员共享的待办数据集合。
- Task：待办任务。
- Gist：Gitee 代码片段（本文统一称“代码片段”），用于承载 JSON 数据文件。
- PAT：Personal Access Token（个人访问令牌），用于调用 Gitee API。
- Sync：数据同步（本地缓存/离线变更与远端代码片段之间的双向同步）。

### 1.5 参考与约束

- 技术栈：Rust + Tauri 2 + Vue 3。
- 外部依赖：Gitee API（v5），网络连接（首次登录/同步时必须）。

## 2. 总体描述

### 2.1 产品视角

MyTodos 是一个无需自建后端的协作待办应用。所有团队数据以结构化 JSON 文件形式存储在 Gitee 代码片段中；客户端通过 PAT 直接调用 Gitee API 完成数据读写，并在本地维护缓存与离线操作队列，实现弱网/离线可用。

### 2.2 用户角色与特征

- 团队管理员（Admin）
  - 创建工作区、生成/配置代码片段、邀请成员加入。
  - 管理工作区基础设置（名称、归档策略、同步策略等）。
- 团队成员（Member）
  - 查看、创建、编辑、完成任务；参与评论与协作。

### 2.3 运行环境

- 移动端：Android / iOS（Tauri Mobile）。
- 设备能力：
  - 本地存储（用于缓存与配置）。
  - 网络访问（HTTPS）。
  - 可选：本地通知（到期提醒）。

### 2.4 设计与实现约束

- 不引入自建后端：所有协作数据只能通过 Gitee API 存取。
- 多端并发写入：需要处理冲突与合并（乐观并发控制）。
- 代码片段容量/速率限制：需控制数据体积、减少全量写入次数、采用增量/分片/归档策略。

### 2.5 关键假设

- 团队会使用“共享的 Gitee 账号 PAT”或“管理员账号 PAT”作为工作区的写入凭证；成员可选择只读或通过同一凭证写入。
- 团队规模 2-5 人，任务量中小规模（建议单工作区活跃任务不超过 2000 条）。

## 3. 功能需求

### 3.1 账户与工作区

**FR-001 Gitee 凭证配置**

- 用户可在应用内配置 Gitee PAT（粘贴输入），用于访问 Gitee API。
- PAT 需支持最小权限集（以满足代码片段读写；实际权限以 Gitee 侧策略为准）。

**FR-002 获取当前用户信息**

- 应用可基于 PAT 获取当前用户基础信息（例如登录名、头像）用于身份展示。

**FR-003 创建工作区**

- 管理员可创建工作区，填写：
  - 工作区名称
  - 工作区描述（可选）
- 创建工作区时，应用应创建或绑定一个 Gitee 代码片段作为远端存储，并生成工作区标识：
  - workspaceId（UUID）
  - gistId（代码片段 ID）
  - workspaceKey（可选：本地加密口令，用于对远端 JSON 加密）

**FR-004 加入/切换工作区**

- 成员可通过“工作区邀请信息”加入工作区：
  - 最小可用：输入 gistId（以及可选的 workspaceKey）。
- 用户可在多个工作区之间切换；切换后列表与缓存应隔离。

**FR-005 成员管理**

- 管理员可维护成员列表（2-5 人）：
  - 添加成员：记录成员显示名、Gitee 登录名（可选）、角色（Admin/Member）
  - 移除成员（仅影响客户端逻辑；由于无后端权限控制，移除仅做“应用内权限/展示”）

### 3.2 任务管理

**FR-101 创建任务**

- 用户可创建任务，字段至少包含：
  - 标题（必填，1-80 字）
  - 详情描述（可选，支持换行）
  - 状态：待办 / 进行中 / 已完成 / 已归档
  - 截止日期时间（可选）
  - 优先级：低/中/高（可选，默认中）
  - 指派人（可选，支持 0-N 个成员）
  - 标签（可选，0-N）

**FR-102 编辑任务**

- 用户可编辑上述字段；编辑后产生一条活动记录（见 FR-140）。

**FR-103 完成/恢复任务**

- 用户可将任务标记为已完成；可恢复为待办/进行中。
- 完成时记录 completedAt、completedBy。

**FR-104 删除任务**

- 用户可删除任务；删除为逻辑删除（soft delete），用于同步与审计：
  - deletedAt、deletedBy

**FR-105 任务列表**

- 应用提供任务列表视图，支持：
  - 按状态筛选（待办/进行中/已完成/已归档）
  - 按指派人筛选
  - 按截止日期筛选（今天/本周/逾期/无截止）
  - 按标签筛选
  - 排序（默认：逾期优先，其次截止时间升序，其次创建时间降序）

**FR-106 搜索**

- 支持按标题/描述关键字搜索（本地索引即可）。

### 3.3 协作能力

**FR-121 评论**

- 任务详情支持评论流：
  - 文本评论
  - 记录 commentId、作者、时间

**FR-122 @成员（可选）**

- 评论中支持选择成员进行提醒（实现为本地通知或标记，不依赖推送）。

**FR-140 活动记录**

- 对关键操作记录活动（Activity Log）：
  - 创建/编辑/完成/恢复/删除
  - 指派变更、截止日期变更、标签变更
- 活动记录用于冲突合并与变更追踪展示。

### 3.4 同步与离线

**FR-201 手动同步**

- 提供“立即同步”入口：拉取远端最新数据并推送本地待提交变更。

**FR-202 自动同步**

- 应用启动后与回到前台时自动同步一次（可在设置中关闭）。
- 在编辑保存时可触发轻量同步（策略可配置：立即/延迟/仅手动）。

**FR-203 离线可用**

- 无网络时仍可浏览本地缓存、创建/编辑任务；变更进入“待同步队列”。
- 网络恢复后可一键或自动重试同步。

**FR-204 冲突检测**

- 当远端数据相对本地基线发生变化且与本地待提交变更冲突时，应用必须检测冲突。
- 冲突场景示例：
  - 同一任务被两端同时编辑同一字段
  - 同一任务一端删除另一端编辑

**FR-205 冲突解决**

- 默认策略：按任务粒度三方合并（base/local/remote），字段级以 updatedAt 进行“最后写入者优先”。
- 无法自动合并时，进入冲突解决界面，允许用户选择保留“本地/远端/手动合并”。

### 3.5 提醒与归档

**FR-301 到期提醒**

- 对设置了截止时间的任务提供本地提醒：
  - 默认在截止前 15 分钟提醒（可配置：关闭/5/15/60 分钟/1 天）

**FR-302 归档策略**

- 支持将已完成超过 N 天的任务自动归档（默认 30 天，0 表示关闭）。

### 3.6 数据备份与迁移

**FR-401 导出备份**

- 支持导出当前工作区数据为 JSON 文件（仅本地导出，不走外部存储）。

**FR-402 导入备份**

- 支持从 JSON 文件导入并覆盖/合并到当前工作区（选择覆盖或合并策略）。

**FR-403 重建工作区存储**

- 管理员可将工作区数据迁移到新的 gistId（例如旧代码片段空间不足）。

### 3.7 设置

**FR-501 同步设置**

- 可配置：
  - 自动同步开关
  - 同步频率（前台：每 N 分钟；后台：仅在回到前台）
  - 冲突解决默认策略（自动/总是询问）

**FR-502 安全设置**

- 可配置是否启用“远端数据加密”（启用后写入远端前加密 JSON）。

## 4. 外部接口需求

### 4.1 与 Gitee API 的交互

- 认证方式：PAT（access_token）随请求携带。
- 主要能力：
  - 创建代码片段（用于初始化工作区）
  - 获取代码片段内容（拉取远端数据）
  - 更新代码片段内容（推送变更）
- 失败处理：
  - 401/403：提示凭证无效或权限不足，引导重新配置 PAT
  - 404：代码片段不存在，引导重新加入或迁移
  - 429/5xx：指数退避重试，保留离线队列

### 4.2 与系统能力的交互

- 本地通知：用于截止提醒、@提醒（可选）。
- 系统安全存储：用于保存 PAT、workspaceKey 等敏感信息。
- 文件系统：用于导入/导出备份文件。

## 5. 界面需求（UI/UX）

### 5.1 全局原则

- 单手可操作：核心操作（新增、完成、切换状态）在 2 次点击内完成。
- 信息层级清晰：列表聚焦任务标题与关键状态（截止/指派/优先级）。
- 弱网提示明确：同步状态应可见（同步中/已同步/有待同步/冲突）。

### 5.2 页面清单与要点

**UI-001 首次进入/引导页**

- 输入/粘贴 Gitee PAT
- 校验成功后进入工作区页

**UI-002 工作区列表/切换页**

- 展示已加入工作区
- 支持“创建工作区”“加入工作区（输入 gistId）”

**UI-101 任务列表页（主页面）**

- 顶部：工作区名称 + 同步状态入口
- 中部：任务列表（按筛选/排序）
- 底部或悬浮按钮：新增任务
- 列表项展示：
  - 标题、截止时间、指派人头像/缩写、优先级标识
  - 复选框/滑动操作完成与删除

**UI-102 任务详情页**

- 展示完整字段：标题、描述、状态、截止、优先级、指派、标签
- 评论区与活动记录
- 支持编辑入口与状态快捷切换

**UI-201 筛选与搜索**

- 支持组合筛选：状态 + 指派 + 截止 + 标签
- 搜索框支持即时过滤

**UI-301 冲突解决页**

- 展示冲突任务及字段差异（本地 vs 远端）
- 支持选择保留版本或手动编辑合并

**UI-401 设置页**

- 同步设置
- 安全设置（PAT 管理、远端加密开关）
- 备份导入/导出
- 关于（版本号、数据存储说明）

## 6. 数据库/数据需求

### 6.1 总体策略

- 远端为事实源（Source of Truth）：Gitee 代码片段中存储工作区数据。
- 本地用于：
  - 缓存（加速列表与离线浏览）
  - 离线队列（待同步操作）
  - 同步状态（版本号、最后同步时间、远端指纹）
  - 用户与工作区配置（敏感信息加密存储）

### 6.2 远端数据结构（Gitee 代码片段）

建议一个工作区对应一个代码片段，包含多个文件（便于分拆与减少冲突面）：

- meta.json：工作区元信息与版本
- todos.json：任务与评论数据
- activity.json：活动记录（可按月归档，例如 activity-2026-05.json）

**DB-REMOTE-001 meta.json（示例字段）**

- schemaVersion：整数，数据结构版本
- workspace：
  - workspaceId（UUID）
  - name、description
  - createdAt、updatedAt
- members：成员数组（memberId、displayName、giteeLogin 可选、role）
- revision：
  - remoteRevision：字符串（远端版本指纹，可用更新时间/commitId/自增号）

**DB-REMOTE-002 todos.json（示例字段）**

- tasks：任务数组
  - taskId（UUID）
  - title、description
  - status、priority
  - dueAt（可空）
  - assigneeIds（数组）
  - tagIds（数组）
  - createdAt/createdBy、updatedAt/updatedBy
  - completedAt/completedBy（可空）
  - deletedAt/deletedBy（可空）
- comments：评论数组
  - commentId（UUID）
  - taskId
  - authorId、content、createdAt

**DB-REMOTE-003 activity.json（示例字段）**

- activities：活动数组
  - activityId（UUID）
  - taskId（可空：例如工作区事件）
  - type（created/updated/completed/…）
  - payload（变更摘要）
  - actorId、createdAt

**DB-REMOTE-004 数据大小与归档**

- 当 activity 文件超过阈值（例如 512KB）时按月份切分。
- 已归档任务可迁移到 archive-YYYY.json（可选），减少主文件体积。

### 6.3 本地数据结构（缓存与队列）

本地可采用 SQLite 或键值存储（实现由技术方案决定），需满足以下逻辑表/实体：

**DB-LOCAL-001 settings**

- giteeAccessToken（加密存储）
- enableRemoteEncryption（布尔）
- workspaceKey（加密存储，可选）
- syncPolicy（自动/手动、频率等）

**DB-LOCAL-002 workspace_cache**

- workspaceId、gistId
- metaJson（缓存）
- lastSyncAt
- lastRemoteRevision（用于冲突检测）

**DB-LOCAL-003 tasks_cache**

- taskId、workspaceId
- taskJson（或字段化存储）
- updatedAt、deletedAt

**DB-LOCAL-004 pending_ops（离线队列）**

- opId（UUID）
- workspaceId
- opType（create/update/delete/complete/…）
- entityType（task/comment/member/…）
- payload（变更内容）
- baseRemoteRevision（提交时基线）
- createdAt、retryCount、lastError

### 6.4 数据一致性与并发控制

- 客户端写入远端时必须携带“基线 revision”，用于判断远端是否已变化。
- 远端已变化时：
  - 先拉取最新远端数据
  - 对 pending_ops 进行重放（replay）并尝试合并
  - 产生冲突则交由 UI-301 处理

## 7. 非功能需求

### 7.1 性能

- NFR-001 启动性能：冷启动进入任务列表不超过 3 秒（中端机）。
- NFR-002 列表性能：1000 条任务滚动流畅，无明显卡顿。
- NFR-003 同步性能：正常网络下，小规模变更同步应在 2 秒内完成（不含网络抖动）。

### 7.2 可靠性与可用性

- NFR-101 弱网处理：网络错误不丢数据，所有本地变更可恢复重试。
- NFR-102 崩溃恢复：崩溃后重启不丢待同步队列与本地缓存。

### 7.3 安全

- NFR-201 敏感信息保护：PAT、workspaceKey 必须使用系统安全存储加密保存。
- NFR-202 传输安全：与 Gitee API 通信必须走 HTTPS。
- NFR-203 最小暴露：应用日志不得输出 PAT、完整请求体中的敏感字段。

### 7.4 兼容性与可维护性

- NFR-301 平台兼容：Android/iOS 适配主流分辨率与深色模式（可选）。
- NFR-302 可迁移：支持将工作区迁移至新的 gistId（对应 FR-403）。

### 7.5 易用性

- NFR-401 核心闭环：新增任务 → 指派 → 完成，路径不超过 3 个页面层级。
- NFR-402 反馈明确：同步状态、失败原因、冲突状态必须可见且可操作。

## 8. 验收标准（摘要）

- 能创建/加入工作区，并在 2-5 台设备间完成双向同步。
- 断网可新增/编辑任务，联网后可自动或手动同步到远端。
- 同一任务多端编辑能检测冲突，并提供可理解的解决界面。
- 数据确实存储在 Gitee 代码片段中，且无需自建后端即可完整使用。
