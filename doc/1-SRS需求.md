# MyTodos 团队待办 App — 需求规格说明书（SRS）

## 1. 引言

### 1.1 文档目的

本文档用于描述“MyTodos 团队待办 App”的需求规格，包括功能需求、非功能需求、界面需求、数据库/数据需求与外部接口需求，为产品设计、开发实现、验收测试提供依据。

### 1.2 产品一句话需求

一个面向 2-5 人小团队的移动端待办事项协作应用。使用 Rust + Tauri + Vue 3 开发，通过 Gitee 代码片段（Gist）API 存储数据，无需自建后端数据存储。

### 1.3 范围与边界

- 目标用户：2-5 人的小团队，如家庭成员，有管理员，生成待办事项的人，处理待办事项的人。
- 目标端：移动端优先（Android/iOS），同时可以编译桌面端（Windows/MacOs/Linux）方便调试，共用一套代码。
- 数据存储：远端使用 Gitee 代码片段 API，实时读写Gitee数据。
- 不在范围内：
  - 自建服务器、传统后端数据库、IM 即时通讯。
  - 复杂的组织/权限体系（例如多部门、多项目层级、细粒度 ACL）。
  - 企业级审计、SLA、可用性承诺。
  - 本地不需要做缓存，离线队列、配置存储等。

### 1.4 术语与缩略语

- Workspace：工作区/团队空间，一组成员共享的待办数据集合。
- Task：待办任务。
- Gist：Gitee 代码片段（本文统一称“代码片段”），用于承载 JSON 数据文件。
- PAT：Personal Access Token（个人访问令牌），用于调用 Gitee API。
- Sync：数据同步（本地缓存/离线变更与远端代码片段之间的双向同步）。

### 1.5 参考与约束

- 技术栈：Rust + Tauri 2 + Vue 3。
- 外部依赖：Gitee API（v5），网络连接（首次登录/同步时必须）。

## 2. 总体描述

### 2.1 产品视角

MyTodos 是一个无需自建后端的协作待办应用。所有团队数据以结构化 JSON 文件形式存储在 Gitee 代码片段中；客户端通过 PAT 直接调用 Gitee API 完成数据读写，不能脱机使用。

### 2.2 用户角色与特征

- 团队管理员（Admin）
  - 创建工作区、生成/配置代码片段、邀请成员加入。
  - 管理工作区基础设置（名称、归档策略、同步策略等）。
- 生成待办事项（Create do Member ）
  - 查看、创建、编辑、完成任务；参与评论与协作。
- 处理待办事项（Todo Member）
  - 查看、创建、编辑、完成任务；参与评论与协作。

### 2.3 运行环境

- 移动端：Android / iOS（Tauri Mobile）。
- PC端口：Windows / MacOS / Linux
- 设备能力：
  - 网络访问（HTTPS）。
  - 可选：本地通知（到期提醒）。

### 2.4 设计与实现约束

- 不引入自建后端：所有协作数据只能通过 Gitee API 存取。
- 多端并发写入：需要处理冲突与合并（乐观并发控制）。

### 2.5 关键假设

- 整个APP内置Gitee帐户，在编译时通过.env读取Gitee帐号信息。
- 团队规模 2-5 人，任务量中小规模（建议单工作区活跃任务不超过 200 条）。

## 3. 功能需求

### 3.1 账户与工作区

**FR-001 Gitee 凭证配置**

**FR-003 创建工作区**

- 管理员可创建工作区，填写：
  - 工作区名称，例如：一中实验初中，软件谷小学等
  - 工作区描述（可选）

- 创建工作区时，应用应创建或绑定一个 Gitee 代码片段作为远端存储，并生成工作区标识：
  - workspaceId（UUID）
  - gistId（代码片段 ID）
  - workspaceKey（可选：本地加密口令，用于对远端 JSON 加密）

**FR-004 加入/切换工作区**

- 成员在启动时需要选择工作区。
- 用户可在多个工作区之间切换。

**FR-005 成员管理**

- 管理员可维护成员列表（2-5 人）：
  - 添加成员：记录成员显示名、角色（管理员，家长，学生）
  - 移除成员

### 3.2 任务管理

**FR-101 创建任务**
注：只有家长角色才能创建任务，管理员、学生也不可以。

- 用户可创建任务，字段至少包含：
  - 标题（必填，1-80 字）
  - 详情描述（可选，支持换行，1-256字）
  - 状态：(必填) 待办 / 进行中 / 已完成 / 已归档 （默认待办）
  - 截止日期时间（必填）
  - 优先级：低/中/高（必填，默认中）
  - 指派人（必选，只支持 1 个成员）

**FR-102 编辑任务**
注：只有家长角色才能编辑任务，管理员、学生也不可以。
- 用户可编辑以上字段；

**FR-103 完成/恢复任务**
注：只有家长和学生角色才能完成或恢复任务，管理员不可以。
- 用户可将任务标记为已进行中，完成；可恢复为待办/进行中。
- 在标记为进行中，需要记录任务开始时间
- 在标记为完成，需要记录任务完成时间

**FR-104 删除任务**
注：只有家长角色才能删除任务，管理员、学生都不可以。
- 用户可删除任务；删除为逻辑删除，标记为删除状态

**FR-105 任务列表**

- 应用提供任务列表视图（家长角色），支持：
  - 按状态筛选（待办/进行中/已完成/已归档）
  - 按指派人筛选
  - 按截止日期筛选（今天/本周/逾期/无截止）
  - 排序（默认：逾期优先，其次截止时间升序，其次创建时间降序）

- 应用提供任务列表视图（学生角色），支持：
  - 默认只显示待办/进行中的任务
  - 右上角有过滤条件（进行中，完成），进行中：只显示待办/进行中的任务，完成：只显示 已完成的任务
  - 排序（默认：逾期优先，其次截止时间升序，其次创建时间降序）


## 4. 外部接口需求

### 4.1 与 Gitee API 的交互

- 认证方式：PAT（access\_token）随请求携带。
- 主要能力：
  - 创建代码片段（用于初始化工作区）
  - 获取代码片段内容（拉取远端数据）
  - 更新代码片段内容（推送变更）
- 失败处理：
  - 401/403：提示凭证无效或权限不足，引导重新配置 PAT
  - 404：代码片段不存在，引导重新加入或迁移
  - 429/5xx：指数退避重试，保留离线队列

### 4.2 与系统能力的交互

- 本地通知：用于截止提醒、@提醒（可选）。
- 系统安全存储：用于保存 PAT、workspaceKey 等敏感信息。
- 文件系统：用于导入/导出备份文件。

## 5. 界面需求（UI/UX）

### 5.1 全局原则

- 单手可操作：核心操作（新增、完成、切换状态）在 2 次点击内完成。
- 信息层级清晰：列表聚焦任务标题与关键状态（截止/指派/优先级）。
- 弱网提示明确：同步状态应可见（同步中/已同步/有待同步/冲突）。

### 5.2 页面清单与要点

**UI-001 首次进入/引导页**
- 首次进入，选择 角色，输入6个数字口令，保存到本地，第二次就直接从本地读取角色
- 通过内置的帐号，访问 Gitee PAT 数据
- 如果是管理员角色，进入后进入工作区配置，可以增删改查。
- 如果是家长角色，进入后选择指定工作区，进入工作区对应的事项列表，可以发布与管理事项。
- 如果是家长角色，进入后选择指定工作区，进入工作区对应的事项列表，只可以查看事项，处理事项（进行中，已完成）。


**UI-002 工作区列表/切换页**

- 展示已加入工作区
- 支持“创建工作区”“加入工作区（输入 gistId）”

**UI-101 任务列表页（主页面）**

- 顶部：工作区名称 + 同步状态入口
- 中部：任务列表（按筛选/排序）
- 底部或悬浮按钮：新增任务
- 列表项展示：
  - 标题、截止时间、指派人头像/缩写、优先级标识
  - 复选框/滑动操作完成与删除

**UI-102 任务详情页**

- 展示完整字段：标题、描述、状态、截止、优先级、指派、标签
- 支持编辑入口与状态快捷切换

## 6. 数据库/数据需求

### 6.1 总体策略

- 远端为事实源（Source of Truth）：Gitee 代码片段中存储工作区数据。
- 本地用于：
  - 保存角色和口令，当前选择的工作区

### 6.2 远端数据结构（Gitee 代码片段）

建议一个工作区对应一个代码片段，包含多个文件（便于分拆与减少冲突面）：

- meta.json：工作区元信息与版本
- todos.json：任务与评论数据

**DB-REMOTE-001 meta.json（示例字段）**

- schemaVersion：整数，数据结构版本
- workspace：
  - workspaceId（UUID）
  - name、description
  - createdAt、updatedAt
- members：成员数组（memberId、displayName、giteeLogin 可选、role）
- revision：
  - remoteRevision：字符串（远端版本指纹，可用更新时间/commitId/自增号）

**DB-REMOTE-002 todos.json（示例字段）**

- tasks：任务数组
  - taskId（UUID）
  - title、description
  - status、priority
  - dueAt（可空）
  - assigneeIds（数组）
  - tagIds（数组）
  - createdAt/createdBy、updatedAt/updatedBy
  - completedAt/completedBy（可空）
  - deletedAt/deletedBy（可空）
- comments：评论数组
  - commentId（UUID）
  - taskId
  - authorId、content、createdAt

### 6.4 数据一致性与并发控制

- 客户端写入远端时必须携带“基线 revision”，用于判断远端是否已变化。
- 远端已变化时：
  - 先拉取最新远端数据
  - 对 pending\_ops 进行重放（replay）并尝试合并
  - 产生冲突则交由 UI-301 处理

## 7. 非功能需求

### 7.1 性能

- NFR-001 启动性能：冷启动进入任务列表不超过 3 秒（中端机）。
- NFR-002 列表性能：1000 条任务滚动流畅，无明显卡顿。
- NFR-003 同步性能：正常网络下，小规模变更同步应在 2 秒内完成（不含网络抖动）。

### 7.2 可靠性与可用性

- NFR-101 弱网处理：网络错误不丢数据，所有本地变更可恢复重试。
- NFR-102 崩溃恢复：崩溃后重启不丢待同步队列与本地缓存。

### 7.3 安全

- NFR-201 敏感信息保护：PAT、workspaceKey 必须使用系统安全存储加密保存。
- NFR-202 传输安全：与 Gitee API 通信必须走 HTTPS。
- NFR-203 最小暴露：应用日志不得输出 PAT、完整请求体中的敏感字段。

### 7.4 兼容性与可维护性

- NFR-301 平台兼容：Android/iOS 适配主流分辨率与深色模式（可选）。
- NFR-302 可迁移：支持将工作区迁移至新的 gistId（对应 FR-403）。

### 7.5 易用性

- NFR-401 核心闭环：新增任务 → 指派 → 完成，路径不超过 3 个页面层级。
- NFR-402 反馈明确：同步状态、失败原因、冲突状态必须可见且可操作。

## 8. 验收标准（摘要）

- 能创建/加入工作区，并在 2-5 台设备间完成双向同步。
- 断网可新增/编辑任务，联网后可自动或手动同步到远端。
- 同一任务多端编辑能检测冲突，并提供可理解的解决界面。
- 数据确实存储在 Gitee 代码片段中，且无需自建后端即可完整使用。
